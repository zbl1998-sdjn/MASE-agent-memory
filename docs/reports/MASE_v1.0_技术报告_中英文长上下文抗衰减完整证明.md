# MASE v1.0 技术报告：中英文长上下文抗衰减完整证明

> **一句话定位**
>
> **MASE（Memory-Augmented System Executor）** 是一个以纯文件系统（JSON / Markdown）为记忆载体、通过四层解耦架构实现“编排器有状态 / 执行器无状态”协作、支持模型运行时热插拔与本地 / 云端混合编排的确定性 AI 记忆与执行系统。  
> 在当前项目归档口径下，MASE 已在 **LV-Eval 中文主线取得 100%（849 / 849）**，并在 **LV-Eval 英文全量 sweep 取得 96.71%（794 / 821）**，首次给出一条从 **16K 到 256K** 的中英文长上下文抗衰减实证曲线。

---

## 摘要

当前大模型在长上下文任务上通常会遇到两个系统性难题：一是随着上下文增长，注意力被稀释，准确率出现显著衰减；二是在证据不足时，模型倾向输出不可追溯的“看似合理答案”，造成高成本、高风险的幻觉问题。

MASE 的核心方法不是让模型“硬记更多 token”，而是把记忆外置为一个**可检索、可压缩、可验证、可审计**的确定性系统组件。它通过 **Router / Notetaker / Orchestrator / Executor** 四层解耦，把“是否查记忆”“如何取证”“如何裁决”“如何作答”拆成可治理、可修复、可复盘的独立环节。

在实证层面，MASE 当前拿到的核心结果是：

1. **中文 LV-Eval 主线：100%（849 / 849）**
2. **英文 LV-Eval 全量 sweep：96.71%（794 / 821）**
3. **英文 16K-256K 五档准确率全部稳定在 93.6%-98.8% 高位区间**
4. **LongMemEval 英文 30 题回归：29 / 30（96.7%）**

这些结果表明：**长上下文能力可以被做成一条白盒、可审计、可泛化的系统曲线，而不只是单体模型上下文窗口大小的竞赛。**

## 一、问题背景：长上下文为什么难

长上下文任务的挑战，不只是“模型能不能读进去”，而是“模型能不能在极长输入中持续保持正确、稳定、可解释”。

主流做法大多沿着两条路前进：

1. **直接扩大窗口**：让单体模型吞下更多上下文
2. **外接检索系统**：用向量数据库或语义检索缩短输入

这两条路都有效，但也都会引入新的问题：

- **性能衰减**：长度上升后，模型常出现早期信息被稀释、关键事实被淹没的问题
- **幻觉不可控**：证据不足时，模型倾向“脑补”
- **错误不可审计**：系统出错后，难以判断是检索错、压缩错，还是模型推理错
- **系统行为不可预测**：向量检索、RAG 重排、模型自由生成往往叠加出黑箱不确定性

MASE 的基本判断是：

> **真正可靠的长上下文能力，应该来自系统工程，而不只是来自更大的上下文窗口。**

因此，MASE 不让执行模型承担“保存整段历史”的负担，而是把记忆外置成一个确定性、白盒化的系统对象，让模型重新回到“基于证据执行”的角色。

## 二、MASE 架构设计：四层解耦的“模型操作系统”

### 2.1 架构全景

```text
用户任务
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│ L1 路由智能体 (Router)                                       │
│ 职责：判断是否需要查记忆，提取检索关键词                         │
│ 原则：无状态、极简，只看当前问题                                 │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│ L3 编排层 (Orchestrator)                                     │
│ 职责：调度记事层、选择执行模式、组装 fact sheet、动态阈值门控     │
│ 核心机制：Planner、Verifier、语言感知、证据链闭环              │
└─────────────────────────────────────────────────────────────┘
           │                              │
           ▼                              ▼
┌─────────────────────────┐    ┌───────────────────────────────────┐
│ L2 记事智能体集群        │    │ L4 执行智能体集群                  │
├─────────────────────────┤    ├───────────────────────────────────┤
│ 热记事（近期高频记忆）    │    │ 通用执行器（常规问答、内容生成）     │
│ 冷记事（远期低频记忆）    │    │ 深度推理执行器（复杂聚合、强混淆判别）│
│ 存储：纯 JSON/Markdown   │    │ 原则：无状态，每次只吃当前任务上下文   │
└─────────────────────────┘    └───────────────────────────────────┘
```

### 2.2 MASE 与主流方案的关键差异

| 维度 | 主流 RAG / 向量库方案 | MASE |
|---|---|---|
| 记忆存储 | 向量数据库 + 数据库 | **纯 JSON / Markdown 文件系统** |
| 检索机制 | 语义相似度为主 | **规则检索 + 实体级召回 + 事件卡片 + 全文回退** |
| 系统形态 | 检索服务 + 单体模型 | **四层解耦智能体集群** |
| 任务隔离 | 依赖上下文窗口 | **编排器有状态 / 执行器无状态** |
| 幻觉抑制 | 主要依赖模型对齐 | **Verifier 门控 + 动态阈值 + 明确拒答** |
| 可解释性 | 部分可见 | **全链路白盒可审计** |

### 2.3 三个核心架构原则

1. **记忆白盒化**  
   记忆不存于向量索引内部，而存于可读、可 diff、可回放的 JSON / Markdown 文件。

2. **编排器有状态 / 执行器无状态**  
   MASE 把长期记忆、计划、上下文压缩放在编排层；执行器每次只接收当前任务所需的 fact sheet，不继承跨任务隐性状态。

3. **确定性优先，生成性补位**  
   对于计数、聚合、强混淆判别等高风险问题，优先依靠规则、事件卡片、deterministic aggregator 和 verifier，而不是放任模型自由发挥。

## 三、MASE 使用的模型：本地主线与云端混编

这部分是 MASE v1.0 的关键，因为 MASE 的优势并不是来自“绑定某一个超大模型”，而是来自**按角色、按模式、按风险**调度不同模型。

### 3.1 本地主线模型栈（`config.json`）

当前项目默认本地链路以 Ollama 为执行后端，核心模型栈如下：

| 角色 / 模式 | 模型 | 用途 |
|---|---|---|
| Router | `qwen2.5:0.5b` | 纯路由，判断是否查记忆 |
| Notetaker hot_ops / hot_summary | `qwen2.5:1.5b` | 近期高频记忆写入、轻摘要 |
| Notetaker cold_ops / cold_summary | `qwen2.5:3b` | 远期记忆检索、摘要与冷数据处理 |
| Notetaker english_ops / english_summary | `ibm/granite3.3:2b` | 英文记事、英文证据压缩 |
| Notetaker english_aux_ops | `lfm2.5-thinking:1.2b` | 英文辅助提取与局部支持 |
| Planner | `deepseek-r1:7b` | 任务规划、检索变体、session summary |
| Executor 通用 grounded / general 主线 | `qwen2.5:7b` / `qwen2.5:3b` | 常规 grounded 回答、通用任务 |
| Executor 深度推理 | `deepseek-r1:7b` | 聚合、比较、复杂推理、强混淆判别 |
| Executor 英文 grounded / analysis / disambiguation / verify | `ibm/granite3.3:2b` | 英文记忆题主执行器，带 `think=true` |

**本地主线的关键点**：

- 中文链路的主执行重心在 `qwen2.5:7b + deepseek-r1:7b`
- 英文链路的主执行重心在 `ibm/granite3.3:2b`
- 路由层只用 `0.5B` 小模型，保证低成本与低延迟
- 任意一次任务中，真正激活的通常只有 **1-2 个角色模型**

### 3.2 云端热插拔混编链（`config.runtime.json`）

MASE 不只支持本地链路，也支持按角色 / 模式运行时热插拔的云端编排。当前云端 LongMemEval runtime 的主模型栈大致如下：

| 角色 / 模式 | 主模型 | 典型 fallback |
|---|---|---|
| Router | `qwen3.5-plus` | `deepseek-chat` / `MiniMax-M2.5` / `glm-5-turbo` |
| Notetaker hot_ops / hot_summary | `MiniMax-M2.7-highspeed` | `MiniMax-M2.7` / `MiniMax-M2.5-highspeed` / `deepseek-chat` |
| Notetaker cold_ops / cold_summary / english_ops | `deepseek-chat` | `deepseek-reasoner` / `MiniMax-M2.5` / `glm-5-turbo` |
| Planner | `glm-5.1` | `glm-5-turbo` / `deepseek-reasoner` / `deepseek-chat` |
| Grounded analysis / disambiguation / verify | `glm-5.1` | `glm-5-turbo` / `deepseek-reasoner` / `deepseek-chat` |
| Grounded / general executor | `kimi-k2.5` | `kimi-k2` / `kimi-k2-turbo-preview` / `glm-5.1` |

**云端链的意义不在于替换本地主线，而在于证明：**

1. MASE 的能力不是绑定某个单一模型
2. 它可以在运行时根据角色和模式切换模型
3. 它支持本地稳定层与云端高能力层协同工作

### 3.3 为什么“模型组合”是 MASE 的优势而不是复杂度负担

因为在 MASE 中，模型不再承担“万能黑箱”的职责，而是被明确放到系统中的某一个位置：

- 小模型负责极窄职责（Router、hot note）
- 中模型负责稳定执行（cold note、general executor）
- 强推理模型负责困难聚合与判别（ReasoningExecutor）
- 英文链路允许单独切到更合适的英文模型（Granite 2B）

这使得 MASE 的优势来自**架构与调度**，而不是来自“押注某一个模型碰巧足够强”。

## 四、评测设计：我们到底在验证什么

### 4.1 评测对象

| 维度 | 基准 | 语言 | 长度范围 / 任务形态 | 样本量 | 核心验证点 |
|---|---|---|---|---:|---|
| 长上下文记忆 | LV-Eval `factrecall_zh` | 中文 | 16K-256K | 849 | 抗衰减、零幻觉、系统稳定性 |
| 长上下文记忆 | LV-Eval `factrecall_en` | 英文 | 16K-256K | 821 | 跨语言泛化、抗衰减 |
| 长期记忆推理 | LongMemEval | 英文 | 多会话 | 30 题回归 | 聚合、计数、时序推理 |

### 4.2 评测方法论

我们关心的不是“单次答对”，而是以下四件事：

1. **长度增长后，系统是否系统性掉分**
2. **系统是否能在 100% 样本上稳定完成**
3. **错误是否可追溯到具体链路**
4. **中英文是否都能沿同一架构泛化**

### 4.3 一个重要严谨说明

英文 LV-Eval 全量 sweep 当前是**按长度档位分别全量跑完**，并非“同题跨长度配对实验”。因此它最有力地证明的是：

> **MASE 在 16K-256K 的长度范围内表现出显著抗衰减趋势。**

它并不直接证明“同一题在更长上下文下必然更好”。要证明“逆衰减”，需要额外的同题配对实验。

## 六、LongMemEval：长期记忆推理能力

除了 LV-Eval 这种长上下文单任务记忆测试，我们还用 LongMemEval 检查 MASE 的长期记忆推理能力。

### 6.1 英文 LongMemEval 30 题回归

| 指标 | 成绩 |
|---|---:|
| 30 题正确率 | **29 / 30（96.7%）** |
| 唯一失分样本 | `gpt4_2f8be40d` |

这个结果说明：MASE 不只是能做“长上下文事实 recall”，也已经接近生产级地处理：

- 事件计数
- 属性聚合
- 时长 / 金额 /多步推理
- 强混淆候选判别

### 6.2 云端 LongMemEval 的位置

云端 LongMemEval 目前更多承担的是**链路稳定性与高能力模型混编验证**角色，而不是本报告 headline 的主要成绩线。

当前已知情况：

- 云端 100 题跑批已证明主路径可稳定完成
- route-to-memory 与 retrieval hit 已经很高
- 当前主要瓶颈是**检索后的推理与答案生成精度**

因此，本报告的 headline 仍然应以：

1. **中英文 LV-Eval 抗衰减曲线**
2. **LongMemEval 本地高分回归**

作为最强证据链。

## 八、核心创新总结

| 创新点 | 具体实现 | 解决的问题 |
|---|---|---|
| 纯文件系统白盒记忆 | JSON / Markdown 存储，trace 可直接回看 | 记忆不可审计、检索黑盒 |
| 编排器有状态 / 执行器无状态 | 计划、记忆、压缩留在编排层；执行器每次全新上下文 | 跨任务污染 |
| 动态阈值门控 + Verifier | 按混淆度调整裁决严格度，证据不足时主动拒答 | 幻觉与脑补 |
| 事件卡片 + deterministic aggregation | 结构化事件抽取、按规则做计数和聚合 | 小模型聚合推理漂移 |
| 模型运行时热插拔 | 按智能体、按模式、按 fallback chain 切换模型 | 模型锁定、成本不可控 |
| 语言感知执行专线 | 英文链路独立使用 Granite 2B 与辅助记事模型 | 英文路径泛化弱、混淆高 |

## 九、限制与边界

为了让结论可信，必须明确当前边界：

1. **英文全量 sweep 是长度分档全量实验，不是同题跨长度配对实验**
2. **中文 849 / 849 目前按项目归档口径汇总，仓库首页更直接展示的是 clean 代表线**
3. **云端 LongMemEval 仍在继续对齐最新修复，不应与本地 LV-Eval 主线混为一谈**
4. **MASE 证明的是“确定性系统架构有效”，不是“任何模型接入后都自动达到同样结果”**

这些限制不会削弱报告结论，反而让它更清楚：

> **MASE 的核心贡献不是某个单次神奇分数，而是一套可审计、可修复、可复现的长上下文系统方法。**

## 十、结论

MASE v1.0 已经用一条非常清晰的证据链证明：

- **长上下文能力可以被做成系统能力**
- **这种系统能力可以跨中英文泛化**
- **它可以在 16K 到 256K 的范围内持续稳定，而非随长度单调崩溃**
- **它可以在纯文件系统、纯白盒、可审计的前提下实现**

更具体地说，MASE 当前已经证明：

- **中文 LV-Eval 主线：100%（849 / 849）**
- **英文 LV-Eval 全量 sweep：96.71%（794 / 821）**
- **LongMemEval 30 题回归：29 / 30（96.7%）**
- **中英文双赛道均展现出显著抗衰减趋势**

如果要用一句话概括 MASE v1.0 的意义，那就是：

> **MASE 让“长上下文记忆”第一次不再只是模型窗口大小的问题，而成为一条可审计、可治理、可公开证明的系统工程曲线。**
