### 主贴正文底稿

我做 MASE，起点很简单：我越来越受不了 AI 记忆系统一旦隐藏到 embedding、chunking 和 reranking 层之后，就很难回答“它为什么会这么答”这种最基本的问题。

所以这个项目选择了更显式的路线：把记忆写成普通文件，把检索和规划拆开，把 fact sheet、候选裁决和 audit trace 都保留下来。

它背后其实还有一个很朴素的灵感：文字和笔记。只有真正被记录下来的信息，才最不容易被遗忘。对人如此，对 AI 也一样。所以 MASE 本质上是在把“记笔记”这件事系统化。

当前仓库里最能直接支撑的结果包括：

- 已验证 clean 切片：
  - `32k-clean`：`10 / 10`
  - `64k-clean`：`10 / 10`
  - `256k-anchor-upgrade (20)`：`20 / 20`
- 在 `64k-clean` 上，本地 **Qwen3.5-27B** baseline：`0 / 10`，在 benchmark 预算内全部 timeout
- 英文 LV-Eval full sweep：16k-256k 共 794 / 821 = 96.71%
- Cloud LongMemEval 100：40 / 100，route-to-memory = 93%，memory route 的 retrieval hit = 100%

对我来说，最有意思的不只是分数本身，而是：当它失败时，我终于能看见它到底失败在哪一步。

### 评论区首楼底稿

大家好，作者在这里。

这个项目来自一个很朴素的痛点：一旦“记忆”变成 embedding + retrieval heuristics + 模型拼接的黑箱组合，很多最基本的 debug 问题都很难回答，比如：

- 系统到底有没有取回正确证据？
- 证据压缩过程中有没有丢失关键事实？
- 模型有没有忽视它已经拿到的证据？

MASE 本质上是我对这个问题的一个白盒化回答。

还有一点我觉得很重要：这并不是一套 2026 年最前沿的大模型阵容，而是一组到今天很多人可能已经会嫌“旧”的 2025 小模型。

这个 repo 现在并不完美。它在 LV-Eval 上比在 LongMemEval 上更强，这一点我不想掩盖。当前公开证据在 memory-heavy recall 上也远强于复杂 coding / 重数学 / 更丰富逻辑任务。但我也正因为这种“不均衡”觉得它有价值——因为它真的把瓶颈位置暴露出来了。

我也越来越相信，系统设计的方向不该是追求一个全能超人，而是让一群各司其职的专业体协作起来。MASE 的多智能体拆分，本质上就是这个思路：这里没有个人英雄主义，只有齐心协力。

欢迎问我架构、benchmark 设定，或者“为什么要用 plain files 而不是 vector-DB-first memory”这些问题。

## 2. Reddit：r/LocalLLaMA 中文底稿

### 标题

**我们用一组 2025 小模型，在已验证 clean 长上下文切片上打到了 0 观察到的幻觉**

### 正文

想分享一个项目，叫 MASE。

它最核心的点，在于它对现在很多 memory tooling 的默认路线是反着来的：

- 记忆不是写进黑盒向量索引，而是外化成 JSON 和 Markdown
- 路由、记事、规划、检索、执行是拆开的
- 系统会保留白盒 trace，而不是把“记忆”简单理解成“vector DB 召回的那几段”

它最初的灵感其实非常日常：记笔记。只有被真实记录下来的信息，才不容易丢失。对 AI 来说也是一样：如果记忆只是模型内部一段会漂移的隐状态，它终究不可靠；只有写下来、留得住、能回看，记忆才真正成立。

我个人最喜欢这个架构的一点是：它更可检查。只要一个 memory question 失败了，我可以直接回看保存的记忆、检索路径、fact sheet 和最终回答行为，而不是只能靠猜。

我也很喜欢它的另一点：它不是在崇拜一个万能模型，而是在尝试构造一群各司其职的专业模型。很多时候，小模型不是不行，而是以前没有被放到对的位置上。

当前仓库里最强的 benchmark 线在 LV-Eval：

- 已验证 clean 切片：
  - `32k-clean`：`10 / 10`
  - `64k-clean`：`10 / 10`
  - `256k-anchor-upgrade (20)`：`20 / 20`
- 英文 full sweep：16k-256k 共 794 / 821 = 96.71%

LongMemEval 这边目前更复杂一些：

- Cloud LongMemEval 100：40 / 100
- route-to-memory：93%
- retrieval hit on memory-routed cases：100%

所以它显然还没有“做完”。架构仍然依赖模型能力，这也是为什么热插拔不是点缀，而是核心功能：一些聚合型 case 仍然会暴露小执行器的上限。但这反而正是白盒设计有价值的地方：它把剩下的失败模式变得可读、可分析。

而且对我来说，这个项目还有一个更大的方向：把能沉淀的东西沉淀成文件，用存力换回一部分算力。也许下一阶段 AI 的效率提升，不只是更大的 GPU，而是算力和存力一起并行。

还有一点非常现实：MASE 最近很多提升来自架构迭代，而不是再开一轮微调训练。对个人开发者和小团队来说，这是一条更轻、更快、更便宜的改进路径。

如果你也关心本地模型栈、可审计记忆，或者关心如何用系统设计替代暴力堆上下文，欢迎交流。

## 3. Reddit：r/MachineLearning 中文底稿

### 标题

**[D] 在白盒 AI memory 系统里，context window 在已验证 clean 切片上已经不再是主要限制**

### 正文

分享一个系统项目 MASE，它在探索一个和主流默认值不太一样的 AI 记忆路线。

和“先全量 embedding，之后再近似召回 chunk”这类典型方案不同，MASE 会把记忆外化成普通文件，并把整条链路拆成路由、记事、规划、检索和执行。

它背后的出发点其实很朴素：只有真正被记录下来的信息，才更不容易被遗忘。换句话说，MASE 在做的事情，本质上是把“笔记”升级成一种可工程化、可审计的 AI memory mechanism。

我觉得它有意思，主要在三点：

1. 失败分析更具体
2. 可以把 retrieval correctness 和 answer-generation behavior 分开观察
3. 从问题到答案的路径会留下可审计证据

另外一个我很关心的点是，这种架构并不崇拜一个无所不能的模型。它更像是构造一个由专业体组成的集群，让不同模型只做自己最擅长的事。我觉得这比追求一个“全能超人”更符合工程直觉，也更符合人类社会中协作创造价值的方式。

当前仓库里的 benchmark 证据主要集中在 LV-Eval，上面的结论比较明确：

- 已验证 clean 切片：
  - `32k-clean`：`10 / 10`
  - `64k-clean`：`10 / 10`
  - `256k-anchor-upgrade (20)`：`20 / 20`
- 英文 full sweep：16k-256k 共 794 / 821 = 96.71%

LongMemEval 目前还更弱一些：

- Cloud LongMemEval 100：40 / 100
- route-to-memory：93%
- retrieval hit on memory-routed cases：100%

我觉得这种“LV-Eval 强、LongMemEval 仍不够强”的割裂反而很有意思。它说明当路由和 retrieval 足够强以后，真正的难点会往证据压缩、聚合推理和执行纪律上迁移，而不只是 retrieval 本身。

也很好奇这里有没有人做过类似这种偏白盒、反 vector-DB-first 的 memory 系统。
