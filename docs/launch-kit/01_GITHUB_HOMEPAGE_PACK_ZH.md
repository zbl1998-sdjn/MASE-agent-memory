## 7. 推荐“为什么做这个项目”文案

建议用这段：

> 大多数长上下文 demo 默认一个前提：只要上下文够大，模型自然会记住。MASE 的立场正好相反：可靠记忆不该靠模型硬背，而该靠系统把记忆外化、压缩、路由和审计。

> 真正值得问的问题不是“模型能塞多少 token”，而是“当上下文变大时，系统还能不能稳定找到、证明并复用正确证据”。

> 我做 MASE 的灵感，其实来自一个非常朴素的事实：文字之所以重要，是因为它把会消失的瞬间变成了可回看的记录；笔记之所以重要，是因为只有真正写下来的信息，才最不容易被遗忘。对人如此，对 AI 也一样。

## 8. 小模型叙事

这是 MASE 最值得打出去的社会传播钩子之一，可以大胆用，但要保持事实边界：

> MASE 建立在一套分工明确的开源小模型栈上。小模型负责路由与记事，更强执行器负责 grounded answering 和 reasoning。重点不是模型崇拜，而是 **系统设计胜过暴力堆上下文**。

这里要把“过时小模型”这件事直接说出来：

> 这不是用 2026 年最前沿的大模型硬堆出来的结果，而是一组很多人到今天可能已经觉得“过时”的 2025 小模型集群。也正因为如此，这个结果才更有传播价值。

同时要把依赖关系讲清楚：

> MASE **不是**在说“模型能力已经不重要了”。仓库里的 LongMemEval 结果恰恰说明：一旦遇到聚合、复杂推理、执行层薄弱的问题，模型能力依然会成为瓶颈。这也是为什么“模型热插拔”不是装饰，而是 MASE 的核心能力。

还有一个很重要的点要直接说：

> MASE 最近很多提升，来自检索、压缩、候选裁决、白盒提示词、热插拔和失败处理这些架构迭代，而不是重新开一轮昂贵的微调训练。对工程实践来说，这意味着更快、更轻、更低成本的改进路径。

建议再补一句，把“多智能体协作”讲透：

> 我们更相信一群各司其职的专业强者，而不是一个无所不能的超人模型。多智能体把任务拆开后，每个模型只做自己最擅长的部分，即使是看起来已经过时的小模型，只要真正被“放开手脚”、放到合适位置，也能展现出极强的任务完成能力。

如果要把这层价值讲得更有思想力度，可以直接用这段：

> 对人类社会而言，我们真正想要的也往往不是一个不可解释、不可制衡的“超人”，而是一群彼此协作、各自专业的强者。历史上任何真正伟大的事情，几乎都不是个人英雄主义的产物，而是强大集群协作的结果。MASE 的方法论也是如此：**这里没有个人英雄主义，只有齐心协力。**

当前仓库里能直接支撑的模型叙事包括：

- `qwen2.5:0.5b`：路由
- `qwen2.5:1.5b` / `qwen2.5:3b`：记事与摘要
- `qwen2.5:7b`：grounded execution
- `deepseek-r1:7b`：规划与深度推理

## 9. 可直接写进 README 的“激进但真实”方法论段落

如果你想把你的世界观更完整地放进 README，可以直接加入这段：

> MASE 试图突破的，不只是某个固定 context window 的数字上限，而是系统对“单一上下文窗口”的依赖。它想解决的是另一个更实际的问题：当上下文越来越长时，我们能不能不用继续忍受长上下文幻觉带来的头痛，而是通过架构把记忆写下来、把证据取回来、把错误查出来。

> MASE 的关键不是复杂，而是简洁。它不是凭空发明一切，而是站在前人的肩膀上，把记忆、检索、规划、验证、工具调用、白盒 trace 这些分散在不同论文和工程经验里的思想重新组合起来。简单不意味着弱，恰恰意味着更容易扩展、更容易替换、更容易审计，也更容易按需定制。

> 它还有一个很朴素但很重要的工程方向：把原本需要反复重算的过程，沉淀成真实存在的本地文件。某种意义上，MASE 用“存力”换回了一部分“算力”。也许下一阶段 AI 的效率提升，不只是更大的 GPU，而是让算力和存力一起并行工作。

> 在工程层面，MASE 已经证明了一件事：通过架构设计，我们可以让模型按角色、按 mode、按降级链工作，也可以在后续请求上通过配置热插拔切换模型。像 Ollama 这样的本地模型后端，也不必永远只是单一路径、单角色的调用方式；当任务被拆开后，同一套模型体系就能承担多角色、多模式的协同工作，并为更高吞吐、更强稳定性和后续并行留下空间。

> 更重要的是，MASE 的每一步都尽量做到可追溯、可审计。白盒化不仅仅是“方便 debug”，它也是一种安全观：只有每一步有迹可循，系统才能被更快改进、更快修复，也才能在关键场景下更值得信任。

## 10. 必须真实写出的缺点与边界

建议明确写出：

1. **当前公开证据最强的仍然是 memory-heavy fact recall 与 LongMemEval 风格任务**，不是所有 AI workload。
2. **复杂 coding、重数学、复杂逻辑等方向，当前仓库还没有系统化公开 benchmark**。这不是因为架构不想做，而是因为个人条件与当前阶段的公开测试范围还没覆盖到。
3. **架构依赖模型能力**。白盒架构可以压幻觉、提稳定性、提可审计性，但它不能凭空抹掉弱执行器的能力上限。
4. **MASE 最强的表述应当是 memory-system breakthrough**，而不是“所有下游任务都被解决了”。
