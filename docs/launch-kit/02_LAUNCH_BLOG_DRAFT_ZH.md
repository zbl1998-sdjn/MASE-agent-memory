## 标题备选

1. **上下文窗口再也不是天花板：MASE 在已验证切片上打到了 0 幻觉**
2. **我们已经做到了：一组 2025 小模型集群打赢了长上下文暴力美学**
3. **不训练更大模型，也能把已验证 clean 线路打到 0 幻觉**
4. **MASE：我们如何把长上下文幻觉打成可审计错误**

## 副标题

在已验证 LV-Eval 切片上，MASE 用可热插拔、可审计的白盒架构和一组 2025 小模型，打出了 100%。

---

在已验证 LV-Eval 线路上，我们已经先把最想说的话做出来了：MASE 做到了 **32k-clean 10/10**、**64k-clean 10/10**、**256k-anchor-upgrade 20/20**，在这些已验证切片上实现了 0 观察到的幻觉 / 0 错答。

而且这不是靠一个 2026 年前沿巨型模型硬堆出来的。它来自一个可热插拔、可审计的白盒架构，跑在一组很多人到今天可能已经觉得“过时”的 2025 小模型上。

这当然不意味着所有任务都已经被解决。但它至少说明了一件事：在一类真正重要的 memory-heavy 长上下文任务上，context window 已经不再是唯一的天花板。

今天大多数 AI 记忆系统，默认都从同一个直觉出发：把上下文窗口做得更大，把更多历史塞回 prompt，然后希望模型能记住真正重要的内容。

我们做 MASE，恰恰是因为我们想反过来做。

我们想要一种不会消失在黑盒索引里的记忆。
我们想要一种出了错之后可以追责、可以复盘的系统。
我们想要一个“为什么它会这么回答”有明确证据链可看的架构。

所以，MASE 没有把记忆理解成“embedding 检索回来的那几段文本”，而是把这件事拆成了明确职责：

- 路由层先判断这次问题是否需要记忆
- 记事层负责写、查、读、压缩外部记忆
- Planner 把模糊任务拆成检索动作
- Executor 只基于给定证据完成回答或推理

这个设计选择，会彻底改变系统的“手感”。

这不仅仅是 benchmark 分数的问题。

它关心的是：**AI 记忆能不能变得可检查。**

它真正想突破的，也不只是某个固定 context window 的数字上限，而是系统对单一上下文窗口的依赖。换句话说，MASE 想解决的是一个更工程化的痛点：**能不能别再让长上下文幻觉，成为所有人都头痛却只能忍受的默认代价。**

## 灵感其实来自写字和记笔记

我后来发现，MASE 最原始的灵感并不来自一个多么宏大的 AI 理论，而是来自两件极其朴素的事情：**文字的诞生**，和**记笔记**。

文字之所以伟大，是因为它让会消失的信息第一次拥有了稳定形态。  
笔记之所以重要，是因为只有真正被记录下来的信息，才最不容易被遗忘。

对人类如此，对 AI 也如此。

如果所谓“记忆”只是模型内部一段漂移的隐状态，那么它迟早会模糊、会丢失、会混淆。只有当信息真的被写下来、可以回看、可以复用、可以校验时，它才算真正进入了系统。

所以 MASE 的很多设计，本质上都只是把“记笔记”这件小事系统化了：  
把信息写下来，把证据找回来，把错误留痕，把过程做成白盒。

## 第一个反常识：记忆应该是可见的

今天很多 AI memory / RAG 系统的默认套路是：

1. 把历史切块
2. 写进向量数据库
3. 召回近似片段
4. 重新塞回模型，希望模型拼出答案

这套方式当然能 work，但它有一个非常难受的工程代价：只要结果一出问题，系统就会变得很难排查。

到底是 retrieval 错了？
还是 chunking 错了？
还是 rerank 错了？
还是模型拿到正确证据也没用好？
还是 prompt 里混进了太多无关历史？

当记忆藏在层层抽象后面时，debug 往往会退化成“讲故事”。

MASE 采取的是一种更老派、也更工程化的立场：

> 如果记忆真的重要，那它就应该是可见的。

在 MASE 里，系统可以保留 JSON 记忆、Markdown 审计日志、fact sheet、候选裁决、调用日志和最终回答 trace。你不仅知道系统答了什么，还能知道它为什么这么答。

这也是 MASE 最核心的价值。

## 第二个反常识：不是大模型包打天下，而是小模型分层协作

我们做的第二个选择，是尽量不把一切工作都压给一个巨型模型。

我们更想让系统把不同事情拆开做。

当前仓库里已经能看到这种分层模型栈：更轻的小模型负责路由和记事，更强的执行器负责 grounded answering 和 reasoning。Qwen 与 DeepSeek 这一类开源模型，在不同层承担不同角色。

这件事有两个好处。

第一，成本更可控。

第二，也是更重要的一点：它会强迫整个架构变得显式。只要路由、记忆操作、规划和执行不再混成一团，你就能分别优化它们，也能更清楚地知道失败来自哪一层。

我们的目标不是证明“小模型天然比大模型强”。

我们的目标是证明：

> **小模型 + 精巧系统设计，可以在重要场景里打赢暴力堆上下文。**

还有一点必须明说：以 2026 年的眼光看，这并不是一套“前沿模型阵容”，而是一组很多人已经会觉得老了的 2025 小模型。

也正因为如此，这件事才更有意思。

如果一组已经不再时髦的小模型，依然能在已验证切片上打出这样的结果，那说明真正起作用的，不只是模型大小，而是架构。

## 第三个反常识：我们更需要专业集群，而不是一个超人

我越来越相信，对 AI 系统来说，真正可靠的方向不是制造一个无所不能的“超人模型”，而是构造一群各司其职的专业体。

多智能体协作的价值，不只是“看起来高级”，而是它真的能提高生产力。  
当路由只负责路由、记事只负责记事、规划只负责规划、执行只负责执行时，每个模型都被放到了自己最适合的位置上。

这也是为什么哪怕是体量更小、甚至已经不那么“新潮”的模型，只要被放到正确职责上，也依然能展现出非常强的任务完成能力。问题很多时候不在于模型“不够强”，而在于我们有没有给它一个合适的位置。

对人类社会而言，其实也是一样。

我们真正更愿意拥抱的，往往是一群彼此协作、各自专业的强者，而不是一个无所不能、不可解释、不可制衡的“超人”。后者带给人的第一感觉，很多时候不是惊喜，而是恐惧。

历史也一再证明，任何非常伟大的事情，几乎都不是个人英雄主义的结果，而是强大集群共同创造的结果。

所以如果要用一句话概括 MASE 的方法论，我仍然愿意沿用那句话：

> 这里没有个人英雄主义，只有齐心协力。

## 第四个反常识：简单架构，反而更容易长大

MASE 并不是一个靠复杂性取胜的系统。

恰恰相反，它的很多力量都来自简单。

它并非凭空诞生，而是站在前人的肩膀上，把记忆、检索、规划、验证、工具调用、白盒 trace 这些本来就存在于不同论文和工程实践中的思想重新编排起来。

从这个角度看，MASE 更像一个“集百家之长之后重新出现”的架构，而不是一个凭空发明出来的怪物。

而简单架构最大的好处就在于：它更容易被扩展，更容易按需定制，更容易被替换，也更容易被审计。

## 今天我们真正敢公开讲的结果

我们不想写那种只说“效果很好”的 launch blog，所以先把我们愿意公开背书的结果写清楚。

- 在已验证 clean LV-Eval 切片上，MASE 已经做到：
  - **32k-clean：10 / 10**，平均 **6.57 s / sample**
  - **64k-clean：10 / 10**，平均 **6.60 s / sample**
  - **256k-anchor-upgrade (20)：20 / 20**，平均 **7.88 s / sample**
- 在受限 benchmark 预算下，对比本地 **Qwen3.5-27B** baseline：
  - **64k-clean：MASE = 10 / 10**
  - **64k-clean：Qwen3.5-27B = 0 / 10**，全部 timeout
- 在英文 LV-Eval 16k-256k full sweep 上，MASE 达到：
  - **794 / 821 = 96.71%**

最锋利但依然真实的说法是：

> 在已验证 clean 线路上，MASE 已经把长上下文幻觉打成了可审计错误，并进一步压到了 0 观察到的失败。

更完整、也更严谨的说法是：

> MASE 在中英文 LV-Eval 上都呈现出明显的 anti-decay trend。

这里措辞要非常谨慎。英文 sweep 属于按长度分档的 full sweep，而不是严格同样本配对实验，所以我们更愿意把它表述成 anti-decay trend，而不是形式意义上的 reverse-decay 证明。

这种克制很重要。

我们宁可严谨，也不愿意只为了传播效果去夸大。

## 这个故事还没有结束：LongMemEval 仍然是未完成战场

这也不是一个“我们已经把一切都解决了”的项目。

在当前仓库可见的 Cloud LongMemEval 100 结果里，MASE 是：

- **40 / 100 overall**
- **93% route-to-memory**
- **100% retrieval hit on memory-routed cases**

这个结果真正有意思的地方，不只是分数本身，而是它暴露出了系统瓶颈的位置。

它意味着：系统很多时候已经能正确判断“该去查记忆”，也已经能成功把对应记忆召回来，但后面仍然会在证据压缩、歧义消解、聚合推理或者执行阶段掉分。

这其实是很有价值的工程信息。

很多项目在 launch 时只展示最漂亮的那张图，尽量把坏消息藏起来。我们更愿意反过来做：正因为 MASE 是白盒系统，所以它能把“哪里还没做好”这件事清楚暴露出来。

MASE 的价值，不在于它已经完成，而在于它让未完成的部分可见。

这里还必须再明确两条边界：

1. **当前公开证据最强的仍然是 memory-heavy fact recall 与 LongMemEval 风格任务**，不是所有 AI workload。
2. **复杂 coding、重数学、复杂逻辑等方向，目前仓库还没有系统化公开 benchmark**。这不是因为我认为架构不能往那边长，而是当前个人条件与公开测试范围还没覆盖到。

## 存力与算力，也许应该并行

MASE 还有一个我个人非常在意的方向：它尽量把原本需要反复重算的过程，转化成真实存在的本地文件。

这不是一个无关紧要的小技巧，而是一种很朴素的工程思想：**把能沉淀的东西沉淀下来，用存力换回一部分算力。**

当记忆、fact sheet、trace、裁决依据都落成可复用文件之后，系统就不必每次都从零开始“重新想一遍”。这会释放出大量本来要消耗在重复推理上的计算资源。

也许接下来 AI 领域真正的加速，不只是更大的算力堆叠，而是让算力与存力一起并行工作。

## 热插拔、稳定性与鲁棒性，本质上也是架构红利

我越来越确信，MASE 最重要的价值之一，是它已经用实现证明：**很多人以为必须靠更大模型硬堆出来的东西，其实可以通过架构设计拿到。**

当前系统已经支持按角色、按 mode、按 fallback chain 进行模型切换；配置更新之后，后续请求可以通过 `reload()` 生效。这意味着它并不是把所有任务永久绑定在一个模型上，而是允许系统在不同任务特征下切换更合适的模型。

这件事非常重要，因为它意味着“模型热插拔”不再是口号，而是一个真正可工程化的能力。

同时也正因为如此，我反而更愿意强调：**MASE 不是在说模型能力已经无关紧要。**

仓库里的结果其实已经说明了另一面：当任务进入复杂聚合、复杂推理或者执行器能力不足的区间，模型能力依然会成为瓶颈。也正因如此，热插拔不是一个“锦上添花”的功能，而是架构必须承认现实之后给出的工程答案。

同样，云端链路里关于 timeout 拆分、指数退避、随机抖动重试、连接池复用、fallback chain 的补齐，也说明了另一件事：系统的稳定性和鲁棒性，很多时候并不是模型参数量直接给的，而是架构是否认真对待失败、延迟、重试和降级。

甚至连像 Ollama 这样的本地模型后端，也不必永远被理解成“单一路径、单角色”的工具。只要架构把任务拆开，同一套本地模型体系就可以承担多角色、多模式的协同工作，并为更高吞吐、更强并行留下空间。

还有一个我觉得很重要、但容易被忽略的现实优势：**MASE 最近的很多提升，来自架构迭代，而不是重新开一轮模型微调训练。**

检索改写、证据压缩、候选裁决、白盒提示词、fallback、热插拔，这些改法在工程上往往比重新训练一个模型更快、更轻、更低成本。这也是为什么我觉得 MASE 的路线更适合个人开发者和小团队持续迭代。

## 为什么这件事不止是 benchmark 游戏

Benchmark 当然重要，但它不是我们做这个项目的唯一理由。

我们真正想要的是一种开发者在日常工具链里也能信任的记忆系统，比如：

- 本地 AI 助手
- coding copilot
- 长周期 agent workflow
- 可审计的企业助手
- 需要证据链的研究系统

也正因如此，MASE 的下一步不应该只是“再多跑几个 benchmark”。

下一步更重要的是把它做成基础设施：

- 通过 MCP 暴露，让 Claude Desktop、Cursor 一类客户端能把 MASE 当作外部记忆脑
- 通过 OpenAI-compatible `/v1/chat/completions` 暴露，让现有前端和框架无需定制集成就能接入

一旦这两步做成，MASE 就不再只是“一个有意思的 benchmark repo”，而会开始变成一个真正可复用的 memory substrate。

## 最后，用一句话概括 MASE

如果必须把这个项目压缩成一句话，那我会这样说：

> AI 记忆不应该是黑盒。

记忆应该能落盘。
检索应该能检查。
失败应该能留痕。
回答应该有证据链。

这就是 MASE 真正想做的事。

不是更大的 context。
不是更多的抽象。

而是更多的结构。
更多的证据。
更多的可审计性。

如果你关心本地模型、白盒系统，或者关心一种真正能被工程师检查的 AI 记忆方式，那这就是我们想做的项目。

这并不是一个我想包装成“天才神话”的故事，恰恰相反，它让我一直对 MASE 保持怀疑，也让我更欢迎大家来批评它、拆解它、修正它。

受限于个人设备和当前公开测试条件，我现在只能先做到这些，还没有把复杂 coding、重数学和更丰富逻辑 benchmark 全面公开跑出来。但也正因为这个架构足够简单，我反而相信它还有很大的延展空间。理论上，它可以继续扩展、继续按需定制，也可以慢慢长成更完整的系统。

所以对我来说，MASE 最重要的总结并不是“我做成了一个什么项目”，而是另外一句更朴素的话：

> 我们应该敢于质疑，也应该敢于为之实践；我们不能把希望完全寄托在别人手上，真正强大的永远是集体力量。

如果要说得更接地气一点，那就是：

> 三个臭皮匠，现在真的也许能赶上一个诸葛亮。
