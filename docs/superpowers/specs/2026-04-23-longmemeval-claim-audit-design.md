# LongMemEval 84.8% Claim Audit Design

## Goal

Audit the repository's published **LongMemEval-S 84.8%** claim and determine whether it is:

1. based on the **official full dataset** or a subset/slice,
2. scored with the **official metric** or an internal/derived metric,
3. described consistently across local files and `origin/main`.

The outcome is a repository state where the published claim is **evidence-aligned** without reducing existing runtime or benchmark capability.

## Scope

### In scope

- Inspect benchmark scripts, result files, docs, and referenced datasets related to LongMemEval.
- Verify sample count, split definition, input variant, and scoring methodology.
- Distinguish official evaluation from internal variants such as LLM-judge, focused-input, rescored, slice-only, or partial reruns.
- Compare local state against `origin/main` and identify mismatches.
- Apply low-risk alignment fixes such as documentation correction, naming clarification, metadata cleanup, or result-label repair.

### Out of scope

- Rewriting the benchmark framework from scratch.
- Deleting research assets unless they are obvious garbage or duplicates.
- Breaking public entry points or removing compatibility layers as part of this audit.

## Constraints

- Preserve existing capabilities and public-facing entry points.
- Prefer clarifying labels and evidence trails over deleting research history.
- Treat README benchmark claims as release-grade statements: they must match the strongest defensible evidence in-repo.

## Audit Questions

1. What exact artifact produced the published **84.8%** number?
2. Was that artifact run on the official full LongMemEval-S set, a focused-input/fullset variant, a 54-case slice, or another subset?
3. Was the score produced by official substring matching, LLM-judge, or a hybrid/rescored pipeline?
4. Do local docs, result files, and scripts describe the same methodology as `origin/main`?
5. If they differ, what is the minimum-change fix that makes the repo honest and internally consistent?

## Evidence Sources

### Primary evidence

- `README.md` and any benchmark summary docs
- `BENCHMARKS.md`
- LongMemEval-related scripts under `scripts/` and `scripts/benchmarks/`
- LongMemEval-related tests under `tests/`
- Result summaries / JSON / logs currently in-repo or generated by the benchmark harness
- Dataset references under `data/` or documented fetch instructions

### Secondary evidence

- `DECISIONS.md`, `CHANGELOG.md`, `RELEASE_NOTES_V2.md`, `v2测试.md`
- recent git history for benchmark-related claims
- `origin/main` tree and tracked result metadata

## Recommended Audit Approach

### Approach A — evidence-first claim audit (**recommended**)

Trace the published number back from README/docs to the generating script, result artifact, dataset cardinality, and scoring rule; then align docs/labels to the strongest defensible claim.

**Why this is recommended**

- Lowest risk to runtime behavior
- Best fit for “capability must not regress”
- Produces an auditable evidence chain instead of a cosmetic text edit

### Approach B — rerun-first benchmark reproduction

Attempt to rerun the relevant benchmark path first, then update docs from fresh outputs.

**Trade-off**

- Strongest evidence if reproducible now
- Higher risk of delay, environment drift, and dataset unavailability

### Approach C — doc-only reconciliation

Read docs and result summaries, then patch wording without tracing full provenance.

**Trade-off**

- Fastest
- Too weak for a disputed benchmark claim

## Execution Design

### Phase 1: provenance trace

Build a chain from published claim → referenced benchmark row → script/result artifact → dataset definition → scoring method.

Deliverable:

- a concrete mapping from each published LongMemEval number to its underlying artifact and method

### Phase 2: dataset cardinality audit

Verify whether the cited result uses:

- official full dataset,
- official split but transformed input format,
- a smaller slice (such as 54 cases),
- or a mixed/filtered evaluation set.

Deliverable:

- an explicit statement of evaluated sample count and why that count exists

### Phase 3: metric audit

Verify whether the reported score corresponds to:

- official substring metric,
- LLM-judge,
- dual-metric reporting,
- or a post-hoc rescored figure.

Deliverable:

- a metric table showing what each number means and whether it is official/comparable

### Phase 4: local vs remote alignment

Compare local repository state and `origin/main` for:

- README benchmark rows,
- benchmark docs,
- benchmark scripts,
- result manifests or summaries.

Deliverable:

- a mismatch list with precise file-level alignment actions

### Phase 5: low-risk remediation

Apply only fixes that improve honesty and traceability without degrading capability:

- tighten claim wording,
- rename ambiguous result labels,
- add methodology notes,
- remove obvious garbage/duplicate benchmark residue if discovered,
- patch metadata references that point to the wrong evaluation basis.

Deliverable:

- repository wording and metadata aligned with actual evidence

## Success Criteria

- We can state whether **84.8%** is full-set official, subset, focused-input, LLM-judge, or mixed.
- We can state whether **61%** is the best official substring-comparable number and from what run.
- README and benchmark docs no longer overstate or blur those distinctions.
- Any fixes preserve current runtime and benchmark entry points.

## Risks and Mitigations

### Risk: result artifacts are incomplete or ambiguous

Mitigation:

- prefer explicit documentation correction over inventing stronger certainty

### Risk: local state differs from remote but neither is fully authoritative

Mitigation:

- use git history and script/result provenance to choose the strongest evidence-backed wording

### Risk: benchmark scripts imply one thing while docs claim another

Mitigation:

- trust executable evidence over marketing language

## Expected Remediation Style

Prefer:

- “84.8% on LongMemEval-S (LLM-judge, focused-input fullset, 424/500)” if true
- “61% official substring best” if that is the strongest official-comparable figure

Avoid:

- publishing a single benchmark number without method/split qualifiers when multiple incompatible scoring paths exist

## Final Output

The audit should end with:

1. a short evidence summary,
2. the exact repository files changed,
3. the corrected benchmark wording,
4. any remaining ambiguity that could only be resolved by a fresh rerun.
