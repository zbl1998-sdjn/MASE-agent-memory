## 为什么和 V1 差距这么大

当前最明确的原因不是“GPU 没用”，而是 **当前 benchmark 实际运行链与 README_V1 出成绩的那条能力线并不等价**。

### 1. 检索层显著变弱

本次 benchmark 实际走的是 `src/mase/__init__.py` 里的 benchmark-compatible runtime。

其中 `BenchmarkNotetaker.search()` 当前实现是：

- 先把 query 拆词
- 再对 `memory_log` 全表扫描
- 用字符串命中计数做排序
- 取前 `5` 条

它**不等于** README_V1 / 发布物料里强调的那条能力链：

- SQLite FTS5 / BM25
- Entity Fact Sheet
- query rewrite
- semantic fallback
- time decay
- 候选裁决表

这会直接导致长上下文问答最关键的“正确证据召回”显著变差。

### 2. 错证据进入后，后续角色会一起答偏

在本次一个中文失败样本里，问题是“被世人广泛推崇为现代物理学奠基人的科学家叫什么名字？”，但实际召回到的 memory rows 是一批**《西游记》片段**。
一旦路由正确、但检索错了，后续：

- notetaker 压缩的是错证据
- planner 规划的是错任务上下文
- executor 基于错 fact sheet 保守拒答

所以当前问题更像是 **“低幻觉但低召回”**，而不是简单的“模型乱编”。

### 3. 角色分工已经跑起来，但中间链路质量不够

本次有效样本日志里已经能看到：

- `router = qwen2.5:0.5b`
- `notetaker = qwen2.5:3b`
- `planner = deepseek-r1:7b`
- `executor = qwen2.5:7b`

说明“角色没有按分工跑”这个问题已被排除。
真正的问题是：**角色之间流转的证据质量，远低于 V1 高分线所依赖的那套检索/压缩/验证链。**

### 4. 中文更差，说明词面检索与证据裁决问题更突出

当前中文结果在 `64k / 128k / 256k` 已接近崩塌，这通常说明：

- query rewrite 不够
- 词面变体覆盖不足
- 证据裁决太弱
- long-context factrecall 对“低词面重合”的适应性差

这和 README_V1 中强调的“反长窗口退化”目标明显不一致。

## 总结

本次测试说明：

1. **内存问题已经压住了**，整套跑批能完成。
2. **角色分工也已经真实参与了**，不是只有一个 executor 在跑。
3. **但当前 benchmark 运行线的核心检索/证据链能力，明显弱于 README_V1 出成绩时的那条 MASE 能力线。**

因此，最准确的表述是：

> **这次结果反映的是“当前仓库 benchmark 运行线”的实际能力，不等同于 README_V1 宣传物料中那条历史高分 MASE 线的能力复现。**
